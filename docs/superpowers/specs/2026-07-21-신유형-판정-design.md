# 공공누리 신유형(제0유형·AI유형) 판정 — 설계 문서

- 작성일: 2026-07-21
- 대상: kogl-classifier 프로토타입
- 근거: `docs/신유형/신유형_판정_설계메모.md`, `[붙임1] 공공누리 신유형 적용 가이드라인.pdf`

## 1. 목표

공공누리 개정('26.1.28)으로 신설된 **제0유형(자유이용)** 과 **AI유형(AI 학습 가능)** 을 프로토타입이 판정·표시하도록 한다. HMC 유형모델 재학습 없이, 이미 추출 중인 SSU 저작물 메타 20항목과 HMC 유형(1~4)을 조합하는 **서비스 후처리 규칙**으로 구현한다.

### 핵심 개념(확정된 결정)
- **적용 범위**: 통합검사 + 검사하기 (둘 다).
- **판정 단위**: **저작물별**. 각 저작물이 제0/제1~4 중 하나의 유형을 가진다. 레코드 대표유형은 계약서 HMC 결과 유지.
- **AI유형 = 직교 boolean**: 제1~4유형에 더해지는 "해당/미해당" 속성(`ai_type_applied`). **제0유형은 AI유형 N/A**(제0유형 자체가 AI 학습 포함 자유이용).
- **수동 확정 = 편집으로 통일**: 자동판정값을 기본으로 제시하고, 관리자가 **기존 [수정] 기능**으로 보정. 신유형 전용 검토/승인 상태머신은 두지 않는다. 이 "자동판정값 제시 + 관리자 보정" 원칙은 **모든 유형에 공통**.

## 2. 판정 규칙

Python 함수로 HF Pipeline에 추가. 저작물 메타(20항목)와 계약서 HMC 유형을 입력받아 저작물별 유형을 산출.

```
resolve_kogl_type(work_meta: dict, hmc_type: str|None) -> {
    resolved_type: "KOGL-0".."KOGL-4",
    ai_candidate: bool,          # 제1~4에 AI유형 병행 후보 여부
    reason: str,                 # 판정 근거(사용한 메타 필드 서술)
    low_confidence: bool,        # 자동 추정 불확실(필드 결측 등) → "확인 권장" 표시용
}
```

판정 순서:
```
1. non_protected_work 가 비보호/만료를 의미 → KOGL-0, reason="비보호(만료)저작물"
2. work_for_hire=예 AND portrait_rights 없음(제3자 초상 미포함) → KOGL-0, reason="업무상저작물, 초상권 없음"
3. 그 외 → HMC 유형(KOGL-1..4) 유지
     · co_author_consent 등 권리자 동의 신호 있음 → ai_candidate=true
4. 필드가 비어(미식별) 판정 근거가 약하면 low_confidence=true
```

원칙:
- 제0유형·AI 후보는 **자동 확정하지 않는다**. 판정값으로 제시만 하고 관리자가 편집으로 확정/변경.
- 근거는 HMC 근거문장이 아니라 **사용한 메타 필드**로 생성.
- 만료 판정은 초기엔 **날짜 계산 없이** `non_protected_work` 신호 기반. 사후 70년/공표 70년 계산은 후속(YAGNI).
- 제0유형이면 `ai_candidate`는 항상 false(N/A).

불리언 해석 헬퍼: `"예"/"true"/"Y"/"해당"/"포함"` 등을 참으로, `"아니오"/"없음"/"미해당"/null/""` 를 거짓으로 정규화(느슨한 매칭).

## 3. 실행 위치 (HF Pipeline `pipeline-server/main.py`)

### 3.1 통합 `/process-combined`
- 이미 저작물별 SSU 20항목을 추출(`map_ssu_to_work_fields`). 각 `works[i]`에 `resolve_kogl_type` 적용 → `resolved_type`, `ai_type_applied`(초기 = ai_candidate), `type_reason`, `type_low_confidence`를 병기하여 `contract_metadata.works[i]`(JSONB)에 저장. 스키마 변경 없음.

### 3.2 검사하기 `/process` (신규 작업)
- 현재 계약서 1건만 SSU 처리. **저작물 파일별 SSU 추출을 추가**한다.
  - 파이프라인이 Supabase Storage에서 해당 contract의 `works` 행 파일을 내려받아 각각 SSU `/api/llm-extract` → `map_ssu_to_work_fields` → `resolve_kogl_type` 적용.
  - 결과를 `works` 테이블 각 행에 저장(§4.3 컬럼).
- 계약서 HMC 유형(레코드 대표)은 기존대로 `contracts.gongnuri_type` 유지.

## 4. 데이터 / 타입 / DB

### 4.1 `src/types/index.ts`
- `KoglType`에 `"KOGL-0"` 추가.
- `KOGL_TYPES["KOGL-0"]` 메타: `label: "제0유형"`, `description: "자유이용"`, `color: "#6B21A8"`(보라, 1~4와 구분).
- `Work`에 필드 추가: `resolved_type?: KoglType`, `ai_type_applied?: boolean`, `type_reason?: string`, `type_low_confidence?: boolean`. **AI유형은 enum이 아닌 별도 boolean**.

### 4.2 통합(rights_checks)
- `contract_metadata.works[i]`(JSONB)에 위 4필드 병기. **마이그레이션 불필요**.

### 4.3 검사하기(works 테이블)
- 신규 컬럼 마이그레이션(`003_work_kogl_type.sql`): `resolved_type text`, `ai_type_applied boolean`, `type_reason text`, `type_low_confidence boolean`. 기존 RLS/트리거 유지.

### 4.4 `src/lib/api/classifier.ts`
- `mapHMCToKoglType`은 **불변**(HMC는 여전히 1~4 반환). 신유형 확정은 Pipeline 규칙 함수로 분리.

## 5. UI

### 5.1 유형·AI 뱃지 (공통 컴포넌트)
- 유형 뱃지: `KOGL_TYPES` 사용 → 제0유형 뱃지 자동 지원.
- AI유형 뱃지: 제1~4유형이고 `ai_type_applied`면 유형 뱃지 옆에 **"AI 해당"**(미해당이면 옅게 "AI 미해당" 또는 미표시). 제0유형은 AI 뱃지 없음.
- `type_low_confidence`면 "자동 추정 · 확인 권장" 보조 문구(모든 유형 공통).

### 5.2 통합 결과 (RightsResultView WorksBrowser)
- 저작물별 우측 패널 상단에 **확정 유형 뱃지 + (제1~4 시)AI유형 뱃지 + 근거(`type_reason`)**.
- **편집**: 기존 저작물 수정 UI에 **유형 드롭다운(제0~4)** 과 **AI유형 체크박스**를 추가(제0 선택 시 AI 비활성). 저장은 기존 JSONB 갱신 경로.

### 5.3 검사하기 상세
- 저작물 목록/우측 패널에 저작물별 유형·AI 뱃지 표기.
- 기존 "공공누리 자동분류 정보 [수정]"과 별개로, **저작물 메타 수정 시 유형·AI유형도 편집** 가능하게(works 컬럼 갱신).

## 6. 검토(=편집) 원칙 — 특수 워크플로우 없음
- 파이프라인은 자동판정값(제0/1~4 + AI 후보 + low_confidence)을 산출해 **기본값으로 표시만** 한다.
- 관리자는 **모든 유형**에 대해 [수정]으로 보정. `review_required` 상태머신·승인 버튼은 두지 않는다.
- 자동판정 불확실은 `type_low_confidence` 보조 표시로만 안내(전 유형 공통).

## 7. 테스트
1. **규칙 함수 단위테스트**(`pipeline-server/tests/test_kogl_type.py`):
   - 업무상저작물+초상권없음 → KOGL-0
   - 업무상저작물+초상권 포함 → HMC 유지
   - 비보호(만료) → KOGL-0
   - 그 외+동의신호 → HMC 유지 & ai_candidate=true
   - 제0유형 → ai_candidate=false
   - 필드 결측 → low_confidence=true
2. 통합 E2E: 저작물별 유형/AI/근거 렌더 + 편집 저장.
3. 검사하기 E2E: 저작물별 SSU 추출 + 판정 + 뱃지 + 편집 저장.
4. 회귀: 기존 유형1~4 판정·화면 무영향.

## 8. 범위·리스크
- **가장 큰 신규 작업**: 검사하기 `/process`에 저작물별 SSU 추출 추가(Storage 다운로드 + 다건 SSU + works 컬럼 저장). SSU가 파일당 수십 초라 다건 시 처리시간↑(HF 600초 내 관리, 필요 시 순차/제한).
- **데이터 품질**: `work_for_hire`·`portrait_rights`·`non_protected_work` 실채움률이 낮으면 규칙이 공회전 → `low_confidence` 표시로 완화, 관리자 편집으로 확정. (SSU 품질 개선은 별건)
- **업체 요청 불필요**: HMC/권리 모델 변경 없음. SSU는 필드 정확도 이슈 시 개선 요청(로직 아님).
